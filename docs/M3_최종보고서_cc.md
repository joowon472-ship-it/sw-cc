#  AI 기반 개인 맞춤형 학습 계획 생성 시스템

## 1. 프로젝트 개요

### 1.1 프로젝트명

**AI 기반 개인 맞춤형 학습 계획 생성 시스템**
### 1.2 배경 및 문제 정의

**대학생의 일정 관리 복잡성:** 대학생들은 과제, 시험, 강의 복습 등 관리해야 할 요소가 많으나 우선순위 설정의 어려움과 즉흥적인 학습으로 인해 전반적인 학습 효율성이 낮음.
**기존 도구의 한계:** 기존의 일정 관리 및 플래너 도구는 사용자가 모든 일정을 수동으로 입력해야 하는 부담이 크며, 계획이 지연되거나 변경될 경우 전체 일정을 재조정하기 어렵다는 문제를 가지고 있음.
### 1.3 목적

**자동화된 학습 계획 수립:** 사용자의 시험 일정과 강의 자료를 기반으로 현실적이고 실현 가능한 학습 계획을 자동 생성함.
**유연한 일정 조정:** 학습이 지연되거나 예기치 못한 일정 변경이 발생했을 때, 시스템이 이를 인지하고 유연하게 계획을 재조정할 수 있는 환경을 제공함.
### 1.4 예상 사용자

시험 및 과제 일정을 체계적으로 관리하고 학습 효율을 높이고 싶은 대학생.
### 1.5 주요 기능 요약

| 번호 | 기능명 | 설명 |
| :--- | :--- | :--- |
| **1** | 회원 관리 | 이메일/비밀번호 기반의 회원가입 및 로그인 기능 |
| **2** | 자료 관리 | 강의 자료 업로드 및 AI를 통한 핵심 내용 요약 제공 |
| **3** | 일정 및 계획 | 과목별 시험 일정 입력 및 이를 기반으로 한 맞춤형 학습 계획 자동 생성 |

 🔄 **M1(중간) 대비 변경 사항:** 시스템의 지속 가능성과 운영 효율성을 확보하기 위해 **관리자(Admin) 기능이 새롭게 추가**되었으며, 이에 따라 사용자 및 자료 관리 기능이 대폭 보강되었습니다.

## 4. WBS 및 프로젝트 일정 (계획 + 실적)

WBS 요약: 본 프로젝트는 기획, 설계, 구현, 검토의 4대 마일스톤을 중심으로 총 14주차 과정에 걸쳐 진행되었습니다.
계획 대비 실적 분석:** * 초기 설계 단계에서 도메인 모델 검토 및 클래스 다이어그램 정밀화 작업으로 인해 예정된 일정보다 **1주 지연**이 발생하였습니다.
지연된 일정을 만회하기 위해 구현 단계에서 핵심 컴포넌트 중심의 집중 개발을 진행하여 개발 속도를 높였습니다.
결과적으로 버퍼(Buffer) 기간을 활용하여 **전체 마감 기한(14주) 내에 모든 태스크를 성공적으로 완료**하였습니다.

## 6. 협업 도구 운영 방식

팀의 생산성 극대화와 투명한 정보 공유를 위해 다음과 같이 도구별 목적을 명확히 분리하여 운영하였습니다.

**Notion (문서화 및 지식 관리):** 회의록, 요구사항 정의서, 설계서 등 모든 공식 산출물 관리.
**GitHub (버전 관리 및 이슈 트래킹):** 코드 형상 관리 및 GitHub Issues를 활용한 주차별 개인 태스크 할당 및 진척도 추적.
**KakaoTalk (실시간 의사소통):** 긴급 안건 공유 및 상시 커뮤니케이션 채널.
 💡 **운영 결과 및 개선:** 프로젝트 초반에는 Notion의 업데이트가 불규칙하게 이루어져 정보 동기화가 늦어지는 문제가 있었습니다. 이를 해결하기 위해 PM이 **매주 월요일 정기 정리 담당자를 명시적으로 지정**하는 규칙을 도입하였고, 이후 최종 단계까지 누락 없는 원활한 문서 관리가 이루어졌습니다.

## 8. 설계 패턴 적용 내역

### 8.1 적용 패턴

**싱글톤 패턴 (Singleton Pattern)**
### 8.2 적용 대상

데이터베이스 연결 관리 클래스 (DBConnectionManager)
### 8.3 적용 이유

웹 시스템 특성상 여러 사용자의 세션이 동시에 생성되는데, 요청이 올 때마다 DB 연결 객체를 개별적으로 생성하게 되면 과도한 시스템 리소스 낭비와 연결 초과 오류가 발생할 수 있음.
이를 방지하기 위해 프로그램 전체에서 단 하나의 DB 연결 인스턴스만 보장하도록 설정하여 리소스를 효율적으로 재사용하고 데이터베이스 접근 안정성을 확보함.

## 9. SOLID 원칙 검토

### 9.1 단일 책임 원칙 (SRP, Single Responsibility Principle) 준수

**기존 문제점:** 초기 설계 당시 LectureMaterial 클래스가 강의 자료의 메타데이터 관리뿐만 아니라, 시스템 알림 기능 및 AI 요약 분석 기능까지 모두 함께 담당하고 있어 하나의 클래스가 너무 많은 책임(변경 이유)을 가지고 있었음.
**리팩토링 내용:** 복잡도를 낮추고 결합도를 줄이기 위해 알림 및 요약 기능을 별도의 Summary 클래스와 NotificationService로 완전히 분리함.
**결과:** LectureMaterial은 자료 자체의 속성 정보만을 관리하도록 책임을 제한하여 SRP 원칙을 엄격히 준수하게 됨.

## 10. 인스펙션 결과

**수행 방식:** 13주차에 팀 내 역할을 서로 교환하여 교차 검증을 수행하는 **Cross-check 방식의 인스펙션**을 전개함.
**발견된 문제 및 조치:** 검토 과정에서 설계 단계의 클래스 다이어그램에 기재된 메서드 명칭과 실제 개발자(Developer)가 구현한 프로토타입 소스 코드 내의 메서드 명칭이 일부 불일치하는 사례를 발견함.
**결과:** 인스펙션 리포트를 바탕으로 즉각 코드와 다이어그램의 용어를 통일하여 시스템의 일관성을 확보함.

## 11. 코딩 표준 문서

팀원 간 코드 가독성 향상과 유지보수 용이성을 위해 아래와 같은 개발 가이드라인을 정의하고 준수하였습니다.

**Naming Convention:**
**클래스명 (Class):** PascalCase 사용 (예: StudyPlan, DBConnectionManager)
**메서드 및 변수명 (Method / Variable):** camelCase 사용 (예: generatePlan(), userId)
**AI 활용 주석 표기:**
개발 효율성을 위해 AI를 통해 생성하거나 뼈대를 잡은 소스 코드 세그먼트의 경우, 코드 최상단이나 해당 라인에 반드시 // AI-generated 주석을 필수적으로 추가하여 코드의 출처와 히스토리를 추적할 수 있도록 함.

## 12. AI 활용 내역 요약

**활용 현황:**
**기획 단계:** 프로젝트 아이디어 구상 및 브레인스토밍 단계에서 대학생 학습 관리 페르소나 정의 시 활용.
**구현 단계:** 복잡한 알고리즘의 예외 처리 로직 설계 및 프로토타입 구현 중 발생하는 코드 오류(디버깅) 탐색에 적극적으로 도입하여 개발 기간을 단축함.
**AI 윤리 및 검증 가이드라인 준수:**
AI가 생성한 결과물을 맹목적으로 수용하지 않고, 반드시 **팀원 전체의 비판적 검토 및 코드 리뷰 절차**를 거침.
발견된 아키텍처적 오류나 비효율적인 구문을 팀의 요구사항에 맞게 최종 수정 및 리팩토링하여 프로덕션 수준의 안정성을 갖춘 코드로 반영함.

## 13. 회고 및 개선 사항

**👍 잘된 점 (Keep):**
GitHub Issues를 기반으로 주차별 스프린트 작업 내역을 명확히 추적한 방식이 주효했음. 각 팀원이 자신의 할 일을 명확히 인지하여 타이트한 일정 속에서도 리스크를 최소화하고 일정을 준수할 수 있었음.
**👎 아쉬운 점 (Problem):**
프로젝트 중반에 관리자(Admin) 요구사항이 추가되면서, 초기 아키텍처 설계를 대폭 수정해야 했음. 이로 인해 충분한 구조적 검토 시간이 다소 부족했던 점이 아쉬움으로 남음.
**💡 배운 점 (Try/Learn):**
설계 단계에서 인터페이스와 데이터 경계를 정밀하게 정의하는 것이 향후 구현 단계의 생산성과 유연성에 얼마나 지대한 영향을 미치는지 온 몸으로 체득함. 향후 프로젝트 시에는 초기 요구사항 변경 가능성을 염두에 둔 확장성 있는 아키텍처 설계에 더 많은 공을 들일 것임.
