### 로그 #1

| 항목 | 내용 |
| --- | --- |
| 날짜 | 2026-04-28 |
| 관련 주차 / 활동 | 8주차 - 기획서 작성 |
| 사용 도구 | Gemini |
| 역할 연관성 | PM으로서 기획서의 핵심 요구사항을 선별하고, 팀원들이 작성한 초안의 기술적 모호성을 검토하여 산출물의 품질을 보증함. |

**프롬프트 (입력한 내용 요약):**

"요구사항 정의서 초안을 작성 중이야. PM 관점에서 '치명적 과실 유형(노이즈, 모호성, 측정 기준 부족 등)'을 기준으로 현재 초안을 비판적으로 분석해줘. 최종 확정 전 단계이므로 수정이 용이하게 보완점을 제시해줘."

**AI 응답 요약:**

> AI는 FR/NFR 항목의 구체성 부족을 지적하며, 특히 NFR 측정 기준 및 FR의 모호한 표현에 대한 개선안을 제안함. 또한 요구사항 정제 과정을 6절 변경 이력과 연동할 것을 권고함.

**비판적 검토:**

> AI의 피드백을 통해 일부 요구사항이 모호함을 확인했으나, 아직 팀 합의가 완전히 이루어지지 않은 '초안' 단계이므로 이를 실제 확정본에 반영하기 전 팀원들과 재논의가 필요함을 인지함.

**팀/개인 수정 내용:**

> 분석가와 논의하여 FR-03 문장에 '업로드한 강의자료를 기반으로'를 명시함. NFR-01은 '사용자가 모든 입력을 완료한 시점부터 10초 이내'로 측정 기준을 명확히 수정함. 또한 FR-01에 이메일 형식 검증 문구를 추가함.

**최종 활용 결과:**

> 팀 내부 검토를 위한 요구사항 정의서 v1.0 초안을 보완하였으며, 향후 수정 이력을 관리하며 완성도를 높여나갈 계획임.

### 로그 #2

| 항목 | 내용 |
| --- | --- |
| 날짜 | 2026-05-18 |
| 관련 주차 / 활동 | 11주차 - M2 설계보고서 작성 |
| 사용 도구 | ChatGPT |
| 역할 연관성 | PM(팀장)으로서 요구사항정의서가 기술적인 시스템 설계(아키텍처, DB)로 누락 없이 반영되었는지 검증하고, M2 마일스톤 제출을 위한 설계보고서의 목차와 개발 일정을 조율하고자 함. |

**프롬프트:**

"대학생을 위한 AI 기반 개인 맞춤형 학습 계획 생성 시스템의 'M2 설계보고서'를 작성해야 해. 요구사항정의서에 정의된 기능(회원가입, 강의자료 업로드, 학습계획 생성 등)을 시스템 아키텍처와 DB 설계로 구체화해야 하는데, 대학생 프로젝트 수준에서 작성해야 할 M2 설계보고서의 표준 목차를 제안해줘. 그리고 이 설계를 바탕으로 구현을 시작할 때 PM으로서 챙겨야 할 개발 일정 마일스톤 예시도 짜줘."

**AI 응답 요약:**

> 대용량 PDF 업로드 시 서버 과부하 리스크, 외부 AI API 연동 시 네트워크 지연 리스크를 지적함.
>
> 프론트엔드와 백엔드 개발을 각각 독립적으로 먼저 진행한 뒤, 프로젝트 마지막 2주 동안 AI API를 연동하고 통합 테스트를 수행하는 4주 단위의 선형적 일정 계획을 제안함.

**비판적 검토:**

> AI의 일정 계획은 우리 프로젝트의 가장 치명적인 비기능 요구사항인 '사용자가 입력을 완료한 시점부터 10초 이내에 학습 계획을 생성해야 한다(NFR-01)'는 성능 기준을 검증할 타이밍을 고려하지 않음.
>
> AI 연동과 속도 최적화를 프로젝트 후반부에 배치할 경우, 성능 타임아웃 문제가 발생했을 때 전체 일정이 마비되는 리스크를 간과함. 또한, 비밀번호 암호화 및 파일 접근 권한 관리(NFR-02)에 대한 보안 테스트 일정도 불명확함.

**팀/개인 수정 내용:**

> AI 연동 및 데이터 처리 속도 최적화 작업을 일정 전반부로 과감히 당겨, 핵심 기능의 프로토타입을 먼저 검증하도록 일정을 전면 수정함.
>
> **4월 28일** 요구사항 도출 단계에서부터 분석가(김연재)의 산출물을 크로스 체크하며 점검했던 이력을 바탕으로, 구현 단계 중간에 '성능(10초 이내 완료) 및 보안(암호화/접근 제어) 집중 테스트' 기간을 별도로 삽입하여 마일스톤의 안정성을 높임.

**최종 활용 결과:**

> AI가 제안한 리스크 요소를 참고하되, 개발 일정의 선후 관계를 팀장 관점에서 비판적으로 재구조화함. 단순 기능 구현 위주가 아닌 '품질 목표(성능/보안)'가 개발 과정에서 실시간으로 검증될 수 있도록 팀원들의 상세 태스크와 마일스톤 일정을 확정하여 M2 설계보고서에 반영함.
